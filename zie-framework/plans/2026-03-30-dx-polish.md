```markdown
## Task Sizing

A good task is completable in one focused session (1–2 hours). Tasks that are
too large lead to incomplete testing; tasks that are too granular create
needless context switching.

### Guidelines by Feature Scope

**Simple features** (1–3 files changed, single pattern):
- Recommended: 2–3 tasks
- Example: add a new enum value, update docstring, bump version
- Fit profile: requires no new abstractions, modifies only existing locations

**Medium features** (4–8 files, new utility functions or refactor):
- Recommended: 5–7 tasks
- Example: new command with help text, argument parsing, basic test coverage
- Fit profile: introduces one new module or class, touches several existing files

**Complex features** (9+ files, new subsystem or architecture):
- Recommended: 8–12 tasks
- Example: new skill with full integration (context bundle, file I/O, error handling)
- Fit profile: introduces multiple new modules, significant refactor of existing patterns

### Maximum Task Count

**Maximum recommended: 15 tasks.** If a feature requires >15 tasks, it likely
represents two independent features or a major subsystem redesign. Split it
into two smaller backlog items with clear dependency ordering.

### Sanity Check

If unsure whether a task is too large, ask yourself:
- "Can I complete this task + write tests + refactor in one focused session (1–2 hours)?"
- "Will the next person easily understand what this task accomplished?"
- "Are there any dependencies between my sub-steps?"

If you answer "no" to the first two questions, split the task.
```
